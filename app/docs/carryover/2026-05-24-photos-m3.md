## 3. User-facing error policy (CRITICAL — supersedes M2)

After review with Jonathan, the 13 error codes M2 shipped are too many.
Most are unreachable from a normal iOS camera-roll pick (the iOS
picker only surfaces photos and videos, so `not-image`,
`unsupported-image`, `is-video`-via-photo-picker, etc. cannot fire).

The new surface has two buckets. **Update `app/src/lib/dispatchErrors.js`
to match.** Keep the internal codes for traceability — only collapse
the user-visible mapping.

### Bucket A — silent and automatic (no UI surface)

User sees nothing except the sync pill incrementing in the album
header. The tile renders immediately from a session blob URL. Queue
drains on the next foreground / online event / successful retry.

| Internal code | Behavior |
| ------------- | -------- |
| `network` | Queue, increment pill, register Background Sync (or fall back per §M4). No toast, no language. |
| `worker-5xx` | Same as `network`. |
| `worker-auth` (expired token) | Attempt silent re-auth. Only surface as a sign-in prompt if re-auth fails. **Re-auth flow is not yet built — for M3, queue silently and document in upload log; don't add UI.** |
| `storage-quota` | Eagerly purge successfully-uploaded queue entries before retry. Warn internally at 80% IndexedDB capacity. Never surface to the user — if the queue genuinely cannot accept the file, log to dev-mode panel and skip the queue insert (don't crash the picker). |
| `missing-file` | Silent no-op (picker returned nothing). |
| `canvas-encode-failed` | Silent retry once. If second attempt still fails, surface per Bucket C (decode-failed copy). |
| `heic-decode-failed` | Silent retry once. If second attempt still fails, surface per Bucket C (decode-failed copy). |

### Bucket C — genuine user-action-required (plain-language UI)

These three messages are the **only** strings that ever appear in
user-visible failure UI. Use them verbatim:

1. **Video exceeds post-encode size cap:**
   > "This video is too long to share. Trim it in Photos first, then share the shorter version."
2. **Photo exceeds post-compression size cap** (rare with 2048px JPEG q=0.85):
   > "This photo is too large. Try sharing a screenshot of it instead."
3. **Photo fails to decode after one silent retry** (includes HEIC):
   > "This photo can't be read right now. Try sharing it again, or share a different photo."

### Banned vocabulary in user-facing copy

No mention of: HEIC, EXIF, codec, queue, IndexedDB, MB, bytes,
compression, encoding, ffmpeg, WebCodecs, mp4-muxer, blob, R2,
Worker, token, auth, sync, drain, retry-loop, attempts, or any other
technical term. Helen's vocabulary is: photo, video, share, trim,
screenshot.

### Backend traceability — CRITICAL

Despite the plain UI, every failure must be fully traceable:

1. **Dev-mode debug log** (in `app/src/lib/uploadLog.js` — new module
   to write): for every silent or surfaced failure, append
   `{ ts, code, message, stack, fileMeta: { name, size, type,
   exifDate }, attempt }` to a ring-buffer in localStorage
   (`rt_upload_log_v1`, capped at ~200 entries).
2. **Queue-entry metadata**: every record in `uploadQueue` already
   has `attempts` + `lastError`; extend to also persist `lastErrorCode`
   so the dev panel can group by code.
3. **Dev settings panel**: add a "View upload log" affordance in
   `app/src/views/Settings.jsx` gated by
   `localStorage.getItem('rt_dev_mode') === 'true'`. Shows the ring
   buffer, code histogram, and a "Copy all" button for paste-into-
   Slack debugging.
4. **Worker observability**: any rejection at the Worker side
   (`POST /assets/photo/:id` or future `/assets/video/:id`) should
   `console.log(JSON.stringify({ ts, traveler, memoryId, kind,
   mime, contentLength, status, error }))` so the entry shows in
   `wrangler tail`. The Worker change for this is non-trivial — if
   touched it needs a user-gated deploy.

Acceptance for §3:
- Helen sees one of the three Bucket C strings or nothing.
- Jonathan (or future Code) can open dev mode, see exactly what
  happened with the technical code and stack, and reason about the
  failure without re-running it.

## 4. M3 scope — video path

### Pipeline
- File picker: `<input type=file accept="video/*">` (separate from the
  photo picker — keep the composer's mode explicit).
- Web Worker (`app/src/workers/encodeVideo.worker.js`):
  - Decode input via `VideoDecoder` (HEVC or H.264 from iPhone).
  - Re-encode via `VideoEncoder` to H.264, 720p (longest edge), bitrate
    ~2 Mbps, keyframe interval ~2 s.
  - Audio: `AudioDecoder` → `AudioEncoder` AAC 128 kbps.
  - Mux via `mp4-muxer` (npm install, ~30 KB gzipped).
  - Post `{type:'progress', percent}` to the main thread continuously.
- Main thread: progress UI in the dispatch modal — large percent
  number + a thin bar. Mandatory; without it Helen will think the app
  froze. Use Fraunces serif for the percent, mono for the bar label.

### Size cap
- Target post-encode: ≤ 25 MB (worker R2 single-PUT cap is plenty
  larger, but 25 MB is a reasonable cellular ceiling).
- If post-encode > cap, surface Bucket C string #1 ("This video is
  too long to share. Trim it in Photos first…"). Do NOT attempt a
  second encode pass at a lower bitrate — keep behavior predictable.

### Queue + retry
- Same shape as photo path. The encoded MP4 blob lands in the queue
  on network failure. The sync pill counts video + photo together.
- The blob URL preview in the album for a pending video should show
  the first frame thumbnail (extract via `VideoDecoder` → first
  decoded frame → `OffscreenCanvas` → JPEG poster) so the tile isn't
  blank during sync.

### Where WebCodecs is unavailable
- Per §3, do NOT surface the WebCodecs-unavailable case to Helen with
  the M2 "Update iOS to share videos" copy. Instead:
  - Pre-detect at modal open (`typeof VideoEncoder === 'function'`).
  - If unavailable, hide the video-picker affordance entirely; show
    only the photo picker.
  - Log the detection result to the upload log so dev mode shows
    "video unsupported on this device" without breaking the user flow.

## 5. M4 scope — Background Sync + offline tests

### Sync strategy
iOS Safari has no Background Sync API. The fallback is two-fold and
runs in parallel:

1. **Service Worker `sync` event handler** in `app/public/sw.js`:
   on `sync` event matching tag `rt-upload-queue`, post a message to
   any active client(s) requesting a drain. Available on Android +
   desktop Chrome.
2. **Page Visibility + interval** in `App.jsx`:
   on `visibilitychange` to `visible`, call `drainQueue(...)`. Also a
   `setInterval` (~120s while open) as a backstop. Universal.

Both paths target the same `drainQueue(runner)` helper that already
exists in `uploadQueue.js`. Idempotent — calling it twice in quick
succession is safe.

### Visible behavior
- Helen comes back to the app after losing signal → sync pill drops
  to zero on its own within seconds.
- No "Drained 3 uploads" toast. No "Background sync registered"
  language. Silent success is the goal.

### Tests
Playwright `context.setOffline(true / false)` is the offline toggle.

- **Offline-pick-then-drain**: go offline → pick a photo → queue
  populates + pill shows → go online → trigger visibility or wait for
  interval → pill drains to zero, tile remains.
- **Foreground-drain trigger**: queue has an item → switch tabs (use
  `page.evaluate(() => document.dispatchEvent(new Event('visibilitychange')))`
  combined with `Object.defineProperty(document, 'visibilityState',
  …)`) → drain fires.
- **Manual sync-pill tap**: covered by M2 already; just regression-
  check it still works.
